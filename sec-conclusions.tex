\section{Conclusions and future work}

\subsection{Limitations}

\begin{itemize}
\item Slow development speed
\item Breaking changes as customizability grows
\item More customizability and sophisticated mechanism lead to
  widening gap between ``aware'' and ``unaware'' user-code such as
  reader macros
\item User-written reader macros can be at odds with many of the goals
  mentioned above (such as no effects, always being able to recover,
  etc.)
\item User-written (s-expression) macros can be at odds with source
  tracking.  The concrete syntax tree library provides
  \texttt{reconstruct} but that is approximate.  Other approaches
  include CST-based or AST-based macros and Scheme-like syntax objects
  but those approaches would be very disruptive for \cl{} users
  (programmers).
\end{itemize}

\subsection{Future Work}

Ideas for future work:
\begin{enumerate}
\item The \cl{} reader associates with characters that occur within
  symbols and numbers the concept of ``constituent traits''.  The
  traits associated with each character are fixed and listed in the
  \cl{} specification.  For \eclector, we plan to implement a ``traits
  table'' that allows clients to change the constituent traits
  associated with each character.

\item We are working on a fully customizable and optimizing
  quasiquotation processor for \eclector{}.  Customizable in this
  context means, for example, that support for quasiquotation can be
  added for new types of literals such as hash-table literals.
  Optimizing means that the processor will turn quasiquoted material
  into forms that can be evaluated efficiently.

\item Literal construction protocol, Quaviver and float overflows

\end{enumerate}
