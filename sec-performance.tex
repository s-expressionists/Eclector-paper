\section{Performance}
\label{sec-performance}

\subsection{General benchmarks}
\label{sec:benchmarks}

As a rough indication of the \eclector{}'s efficiency when reading
\cl{} code, we measured the time it took the native readers of
different \cl{} implementations as well as extrinsic \eclector{} and
\eclector{} configured to produce parse results in those same
implementations to read the contents of \cl{} files taken from
real-world open source libraries.  Code similar to the following was
used to perform the benchmarking:%
%
\begin{listing}[H]
\begin{minted}[fontsize=\small,linenos]{lisp}
(let ((content (alexandria:read-file-into-string filename)))
  #+sbcl (sb-ext:gc :full t)
  #+ccl  (gc)
  (the-cost-of-nothing:benchmark
   (with-input-from-string (stream content)
     (loop for form = (<<read>> stream nil stream)
           until (eq form stream)))))
\end{minted}
\end{listing}
%
\vspace{-2em} \noindent where \texttt{<<read>>} is either
\texttt{cl:read}, \texttt{eclector.reader:read} or
\texttt{eclector.concrete-syntax-tree:read} depending on the
benchmark.

We performed the benchmark described above for \sbcl{} and \ccl{} with
default optimization settings\footnote{Unfortunately, results for ECL
  are missing since the ECL version packaged for the operating system
  of the authors has a
  \href{https://gitlab.com/embeddable-common-lisp/ecl/-/issues/802}{bug}
  that prevents it form executing \eclector{}.  Building ECL from
  source failed due to issues with the default C compiler on the
  author's system.}.  We arbitrarily selected the following large
\cl{} files as inputs for the benchmark runs: from the library
\texttt{flexi-streams-20241012-git} the file \texttt{enc-cn-tbl.lisp}
with $48,418$ lines and from the library \text{screamer-20210807-git}
the file \texttt{screamer.lisp} with $7,233$ lines.  With \eclector{}
commit \texttt{b2b22cdf} we obtained the following results on an
\texttt{Intel(R) Core(TM) i7-4610M CPU @ 3.00GHz} machine with 15 GB
of memory:

\vspace{1em}\noindent%
\begin{tabular}{lrr}
  \textbf{Reader}         & \textbf{enc-cn-tbl.lisp} & \textbf{screamer.lisp}\\
  \sbcl{} native          & 0.034 s                  & 0.011 s\\
  \sbcl{} \eclector{}     & 0.161 s                  & 0.052 s\\
  \sbcl{} \eclector{} CST & 0.475 s                  & 0.096 s\\
  \ccl{} native           & 0.186 s                  & 0.047 s\\
  \ccl{} \eclector{}      & 1.249 s                  & 0.607 s\\
  \ccl{} \eclector{} CST  & 4.108 s                  & 0.961 s\\
\end{tabular}
\vspace{1em}

\noindent We observe that, in a given \cl{} implementation, the
respective native reader takes 8 \% to 20 \% of the time taken by
\eclector{} to read the same input.  On the other hand, \eclector{} on
\sbcl{} is about on par with the native reader of \ccl{}.  When
producing parse results, \eclector{} takes significantly more time, so
that the native readers read the same input in 5 \% to 11 \% of the
time.

\subsection{Pathological inputs}
\label{sec:pathological-inputs}

As a second indication of performance characteristics, we have chosen
two pathological inputs, namely
\texttt{\#1=(\#2=(\#3= \dots \#$N$\#=(\#1\# \#2\# \#3\# \dots \#$N$\#)))}
which we call ``input 1 of size $N$'' and
\texttt{\#1=(\#2=(\#3= \dots \#$N$=(1 2 3 \#$N$\#) \dots \#3\#) \#2\#) \#1\#)}
which we call ``input 2 of size $N$''.  The benchmark consists of
reading the two inputs for different sizes $N$ with \sbcl{}'s reader
and with \eclector{}, and measuring the respective runtimes.
Figure~\ref{fig:runtimes-pathological} shows the runtimes for the
different cases.  For \sbcl{}, the runtime is quadratic in the size
$N$ of the input for both input shapes.  For \eclector{}, the runtime
is linear in the size of the inputs.

The reason for the difference is that \sbcl{} performs the fixup
operation immediately on the return value of each finished
\texttt{read} call.  Since each result contains the previous results,
this strategy leads to an amount of work proportional to
$\sum_{1 \le i \le N} i \propto N^2$.  \eclector{}, on the other hand,
delays the fixup operation until the outermost \texttt{read} call
returns which leads to an amount of work proportional to just $N$.
