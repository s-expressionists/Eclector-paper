\section{Eclector: A highly configurable reader}
\label{sec-eclector}

In this section, we discuss features and other characteristics of
\eclector{}.  \eclector{} was initially created as the native reader
for the \sicl{}%
\footnote{\url{https://github.com/robert-strandh/SICL}} project, still
under development.  From the start, it was designed, in addition to
providing the reader for \sicl{}, to make it possible to parse an
editor buffer as described in a 2018 ELS paper
\cite{durand:hal-01887230}.  Early on, the code was extracted from
\sicl{} into a separate project and renamed to \eclector{}.  From then
on, it evolved entirely independently of \sicl{}, through the work of
the first author of this paper and the project goals became more
clearly defined as well as broader in scope:

\begin{enumerate}
\item \eclector{} should be written in portable \cl{}.
\item \eclector{} should provide a reader that strictly conforms to
  the \cl{} specification.
\item \eclector{} should provide distinct condition types with precise
  and localizable error messages for all possible syntax errors.
\item For invalid inputs, \eclector{} should be able to recover from
  any syntax error and produce a result that represents the invalid
  input as closely as possible if the corresponding restart is
  invoked.
\item All aspects of \eclector{} should be extensible and customizable
  beyond what \cl{} offers by means of special variables and
  readtables.
\item \eclector{} should provide source location tracking and parse
  result construction, including the representation of \emph{skipped
  material} \seesec{sec:source-information}.
\item \eclector{} should achieve acceptable performance in general
  \seesec{sec:benchmarks} and non-catastrophic performance degradation
  for unusual and pathological inputs such as deeply nested or highly
  circular expressions where possible
  \seesec{sec:pathological-inputs}.
\end{enumerate}

After about a decade of slow-paced development in total, \eclector{}
is now a relatively mature project which mostly meets the above goals
and is made available under the ``2-clause'' BSD license with
extensive documentation, many releases and several users.  The
codebase consists of around $5.000$ lines of implementation code and
around $5.000$ lines of test code.  The test suite achieves almost
full line- and branch-coverage of the codebase and applies random
testing techniques for certain aspects of the implementation.

The following sections discuss features that distinguish \eclector{}
from other \cl{} readers (\refSec{sec:features}) and describe some of
the protocols for \eclector{} provides for customization
(\refSec{sec:protocols}).

\subsection{Features}
\label{sec:features}

\subsubsection{Source location information}
\label{sec:source-information}

\eclector{} provides source location information to modules that can
benefit from it such as error conditions and parse results (see
below).  By default, the source location information consists of the
stream position, but clients can define a method on the relevant
generic function to implement some other representation.  For example,
in \sicl{}, the source information consists of the entire input file
in the form of a vector of lines, and the line and column of the start
and the end of the S-expression.

\subsubsection{Parse results and skipped input}

In addition to the return values of reader functions defined by the
\cl{} specification, \eclector{} can produce \emph{parse results}
which can capture more information about the input: each input
expression, even atoms, can be represented as unique objects so that
source location information can be attached unambiguously.
Furthermore, \emph{skipped material} can be represented as parse
results.  By skipped material we mean input which a standard reader
skips, such as comments and inapplicable reader conditionals as well
as input that a standard reader processes automatically without
providing access to the original input such as read-time evaluation,
labeled object definitions and references, and the feature expressions
in reader conditionals.  Clients can configure if and how parse
results are represented.

While parse results and the representation of skipped material serve
little purpose when a reader is used as the first step of a
\texttt{read-eval-print} loop, they are essential when the reader is
used to parse the contents of an editor buffer.

\subsubsection{Specific error condition types}

For each error situation, \eclector{} signals a condition of a type
that is specific to that situation.  That way, client code can handle
each such situation differently.  \eclector{} uses the
acclimation\footnote{\url{https://github.com/robert-strandh/acclimation}}
library to enable error messages in multiple languages but messages
for languages other than English have not been added at this time.

\subsubsection{Recovery from errors}

For every error situation, restarts are available that allow client
code to indicate to \eclector{} that it should recover from the error.
For example, if a (specific subtype of) \texttt{end-of-file} condition
is signaled, recovery consists of acting as if the input is complete,
so that string literals, lists, vectors, etc., are closed.  If the
name of a non-existing package is used as a package prefix for a
symbol, a restart can be invoked that returns an uninterned symbol
instead.

\subsubsection{Unusual inputs and robustness}

While \eclector{} will likely be used for reading source code in most
applications, the possibility of using \eclector{}
\emph{intrinsically} requires it to handle any kind of input the user
of a \cl{} implementation could apply \texttt{read} to.  As one
example, for data serialized as S-expressions, the lengths of lists
and vectors, the nesting depths of expression, the frequency of
circular structure and the overall size of the input can be vastly
different from source code.

To account for this requirement, \eclector{} is designed to avoid
unbounded recursion where possible.  Instead, functions that could
cause unbounded recursion if expressed with naive recursion, initially
process the input recursively but track the recursion depth and employ
iteration and a worklist in case the recursion depth exceeds a
threshold.  However, not all recursion in \eclector{} is handled using
this approach at the moment.  In particular, \texttt{read} calls
cannot nest to an arbitrary depth with the current design.  It may be
possible to leverage the reader state protocol
\seesec{sec:reader-state-protocol} to break up these nested calls into
chunks manageable by recursive processing but we have not attempted
this technique.

\subsubsection{Intrinsic and extrinsic use}

\eclector{} can be loaded into an existing \cl{} implementation
without affecting the native reader, or any other system-specific
code.  It will then define several \eclector{}-specific packages that
a client must refer to in order to use \eclector{}.  We call this way
of using \eclector{} ``extrinsic''.

But \eclector{} can also be used as the native reader of a new \cl{}
implementation, or to replace the existing reader of an existing \cl{}
implementation.  Its main functions will then be defined in the
\texttt{common-lisp} package.  We call this way of using \eclector{}
``intrinsic''.  Intrinsic use in a new \cl{} implementation will
require some specific bootstrapping consideration, since
\clos{} is used extensively both in the implementation of \eclector{}
and for configuring it.

\subsection{Protocols provided by \eclector{}}
\label{sec:protocols}

The possibilities for customization that \eclector{} provides are
realized in the form of a number of protocols.  This section
presents some of the noteworthy protocols\footnote{A complete
  description of the protocols \eclector{} provides for customization
  can be found in the
  \href{https://s-expressionists.github.io/Eclector/eclector.html\#External-protocols}{``External protocols''}
  section of the manual.}.

\subsubsection{The client parameter design pattern}
\label{sec:client-parameter}

For providing an optional and unintrusive customization mechanism to
clients of the library, \eclector{} uses the ``client parameter''
design pattern which is employed in the same way in other libraries
related to the \sicl{} project as well%
\cite[Section 3.1]{DBLP:conf/els/StrandhD22}.  In short, the pattern
consists of adding an additional parameter, usually called
\texttt{client} and positioned as the first element of the lambda
list, to generic functions that should be customizable.  The library
is structured such that a client (user) of the library can easily
arrange for the relevant generic functions to be called with a
particular value for the \texttt{client} parameter.  This organization
allows clients to define methods specialized to the class of the
client-supplied value on the customizable generic functions and have
those methods be the most specific and thus in charge of the behavior
of the respective generic function.  \eclector{} provides a special
variable named \texttt{eclector.base:*client*} which users of the
library can set or bind to a client object.  Functions like
\texttt{eclector.reader:read} pass the value of the variable to
customizable generic function.  As a result, the high-level reader
interface is unchanged compared to the \cl{} specification but aspects
of the reader behavior can be customized by defining methods and
binding a special variable around calls of the high-level reader
function.  Listing~\ref{lis:customization} shows a very simple example
of this customization technique.

\begin{listing}
\begin{minted}[fontsize=\small,linenos,escapeinside=||]{lisp}
(defclass my-client () ())
(defmethod eclector.reader:interpret-token
    ((client my-client) (stream t) (token t) (escape-ranges t))
  (let ((result (call-next-method)))
    (if (typep result 'number) (- result) result)))
(let ((eclector.base:*client* (make-instance 'my-client)))
  (eclector.reader:read-from-string "(5 #C(6 7))"))
|$\Rightarrow$| (-5 #C(-6 -7)), 11
\end{minted}
\caption{A simple customization that makes the reader negate any
  number tokens (\texttt{\#C} is a reader macro, so only the parts are
  affected) before returning them to the caller.}
\label{lis:customization}
\end{listing}

\subsubsection{Reader behavior protocol}
\label{sec:reader-behavior-protocol}

The \cl{} specification describes the core of the reader,
i.e. excluding reader macros, in terms of a ``reader algorithm'' which
employs concepts such as toplevel and recursive \texttt{read} calls,
skipping and preserving whitespace, interpretation of tokens,
invocation of reader macros and more.  However, only a subset of these
concepts and algorithmic aspects are reflected in the programming
interface of the \cl{} reader, namely the functions \texttt{read},
\texttt{read-from-string}, \texttt{read-preserving-whitespace} and
\texttt{read-delimited-list}. \eclector{} tries to promote the
mentioned concepts and algorithmic aspects to first-class status by
providing protocol functions such as \texttt{call-as-top-level-read},
\texttt{read-common}, \texttt{read-maybe-nothing},
\texttt{interpret-token} as well as more specialized protocols, some of
which will be discussed in subsequent sections.  As demonstrated in
Listing~\ref{lis:customization}, these generic functions allow clients
to customize the basic behavior of the reader.

\subsubsection{Reader state protocol}
\label{sec:reader-state-protocol}

Any native \cl{} reader (and the intrinsic version of \eclector{} does
this as well) consults several special variables in order to determine
actions in various situations.  These special variables are set by
user code, by other features of the \cl{} system, and (rarely) by the
reader itself.  For example, the reader uses the special variable
\texttt{*readtable*} to determine which readtable to use to determine
the \emph{syntax type} associated with a character.  Another example
of such a special variable is \texttt{*read-base*} that determines the
numeric base for reading numbers.

When \eclector{} is used extrinsically, it is clearly not possible to
use the host versions of these variables.  For that reason, among
others, \eclector{} defines a \emph{reader state protocol} that
removes the direct use of these special variables.  Instead, the
reader state protocol uses \emph{symbols} (often the same symbols that
name the host special variables) called \emph{aspects} to denote the
variables.

The reader state which \eclector{} queries and manipulates through the
reader state protocol also contains aspects that do not denote any of
the standard special variables: one aspect controls whether the
consing dot is legal in the current state.  Another aspect tracks the
quasiquotation nesting level.  User code can in principle add
additional, custom aspects.  The intention of the reader state
protocol is to allow representing the entire state of the reader as
one or more first-class objects.

The reader state protocol consists of a small number of generic
functions like \texttt{state-value}, \texttt{(setf state-value)} and
\texttt{call-with-state-value} that are called by \eclector{} and
client code to query or modify the aspects of the reader state.  Each
of these generic functions will have an aspect as one of its
arguments.  These functions also have a \texttt{client} parameter that
can be used by client code to create specific methods as explained in
\refSec{sec:client-parameter}.

\subsubsection{Labeled objects}
\label{sec:labeled-objects}

The \cl{} standard specifies the reader macros \texttt{\#N=expression}
and \texttt{\#N\#} for defining and referencing labeled objects
respectively.  We call $N$ the \emph{label}.  The standard describes
the processing of these labeled objects in terms of the observable
results and some constraints.  \eclector{} implements this description
by processing definitions and references for each label $N$ according
to the state machine illustrated in Figure~\ref{fig:labeled-objects}
and a corresponding protocol of first-class objects and operations
(discussed below).

\begin{figure*}
  \centering
  \includegraphics[align=t,width=.60\linewidth]{figures/labeled-object-states.pdf}
  \hspace{-7em}
  \includegraphics[align=t,width=.50\linewidth]{figures/labeled-object-api-states.pdf}
  \caption{Labeled object state machine.  The boxes with rounded edges
    depict the possible states of a label $N$.  A Lisp object
    associated with the label in a given state is shown in the lower
    compartment.  The state machine for each label starts in the
    \texttt{undefined} state before any input has been consumed and
    terminates at the end of input (since new references could appear
    at any point until then). \emph{Left}: Full set of states with
    associated inputs. The edges represent state transitions that
    occur when input which resembles the label next to the edge is
    consumed.  Note that the edge labels show all input consumed so
    far for the respective transition, not the new input compared to
    the origin state.  \emph{Right}: Reduced set of states used in the
    protocol with edges labeled by corresponding protocol functions.}
  \label{fig:labeled-objects}
\end{figure*}

Each label starts out as \emph{undefined} and potentially transitions
to \emph{defined}, \emph{circular}, \emph{final} and
\emph{referenced}.  Based on this state machine, \eclector{} provides
a customizable protocol for processing labeled objects with generic
functions like \texttt{note-labeled-object},
\texttt{forget-labeled-object}, \texttt{reference-labeled-object}.

The second part of handling labeled objects is processing the objects
produced by the reader before they are returned to the caller such
that labeled object references%
\footnote{This delayed substitution is required only in case of
  circular structure.  In cases like
  \mintinline[escapeinside=||]{lisp}{(|\#|1=:a |\#|1|\#|)}, the
  referenced object can be substituted immediately.}  are replaced by
the respective referenced object.  We refer to this process as
\emph{fixup}.  \eclector{} uses a \emph{fixup graph} data structure to
manage the algorithmic complexity of pathological inputs
\seesec{sec:pathological-inputs} as well as another protocol to make
the fixup processes extensible and customizable.  In particular,
clients can add support for fixing up objects such as hash-tables that
don't have a literal syntax in the standard, or
implementation-specific objects.  The fixup protocol consists of
generic functions for controlling the traversal of the fixup graph
like \texttt{walk-fixup-tree} as well as functions for controlling the
substitution in individual objects such as \texttt{fixup} and
\texttt{new-value-for-fixup}.
