\section{Existing \cl{} readers}

In this section, we give an overview of the native readers of some
common free or open-source existing \cl{} implementations.  We focus
on behavior in error situations and features like tracking of source
positions that can be essential in various use cases, particularly
cases where the reader is not used as the first phase of a
\texttt{read-eval-print} loop such as processing the contents of
editor buffers as illustrated in Listing~\ref{lis:bad-program}.  We
list the responses to invalid inputs not as a remark on the quality of
the respective implementation but instead to evaluate whether those
native readers could be used to read (a heuristic approximation of)
invalid inputs as is required in certain use cases.

For the behavior reported here as well as for the benchmarks described
in \refSec{sec-performance}, the following implementations and
versions were used:

\vspace{1em}\noindent%
\begin{tabular}{ll}
  \textbf{Implementation}          & \textbf{Version}\\
  Embeddable Common Lisp (\ecl{})  & ECL 24.5.10\\
  Steel-Bank Common Lisp (\sbcl{}) & 2.6.0.debian\\
  Clozure Common Lisp (\ccl{})     & 1.13 (v1.13-21-gda7138ef)\\
                                   & LinuxX8664
\end{tabular}
\vspace{1em}

Like \cl{}, the \scheme{} programming language employs a reader for
turning textual source code into S-expressions.  However, unlike
\cl{}, modern versions of \scheme{} include the concept of
\emph{syntax objects} which are \scheme{} datums annotated with
additional information such as scope information for identifiers or
source information for forms read from files.  While some of the
challenges presented here can be addressed with syntax objects, they
require a particular interface for the reader and, more importantly, a
different mechanism for macros than \cl{} provides.  Because of those
differences, we consider \scheme{} and syntax objects out of scope for
the following short survey.

\subsection{Embeddable Common Lisp (\ecl{})}

\ecl{} is implemented as a core written in \clanguage{}.  It is built
by first compiling this core and then loading \cl{} code into the
system created form the core.  For that reason, the reader must be
written largely in \clanguage{}.

When an attempt is made to read an incomplete expression, so that the
\texttt{end-of-file} condition is signaled, only the restart
\texttt{restart-toplevel} is available.  Similarly when an attempt is
made to read a symbol with an explicit package marker where the
corresponding package does not exist, an error of type
\texttt{simple-error} is signaled, and only the restart
\texttt{restart-toplevel} is available.  When an attempt is made to
read an incorrect token in the form of a malformed symbol with three
package markers, an error of type \texttt{simple-error} is signaled,
and the message is cryptic.  For example, an attempt to read the
malformed symbol \texttt{cl:::car}, results in the error message
``There is no package with the name \texttt{CL:::C}''.%
\footnote{We will call these three invalid inputs ``\emph{incomplete}
  input'', ``\emph{missing package} input'' and ``\emph{triple colon}
  input'' respectively in the following sections.}

One might think that the reason \ecl{} proposes so few restarts is
because it might be difficult to do so for a function written in
\clanguage{} but that is not the case; \ecl{} has full support for
functionality analogous to \texttt{restart-case} for \clanguage{}
code.

The \ecl{} reader is not involved in source tracking.  The compiler
tracks the source only of top-level forms, so recursive calls to the
reader need not be handled.

\subsection{Steel-Bank Common Lisp (\sbcl{})}

The \sbcl{} reader is written in highly optimized \cl{}, with
strategies like sophisticated management of buffers, elision of
type-checks and bounds-checks, and without the use of the specific
features of the \cl{} Object System (\clos{}) such as standard classes
and generic functions (which are relatively slow compared to ordinary
functions in \sbcl{}).

For the \emph{incomplete} input, the \texttt{end-of-file} condition is
signaled and no restart other than \texttt{retry} and \texttt{abort}
is available.  For the \emph{missing package} input, an error of type
\texttt{simple-reader-package-error} is signaled.  Several restarts
are then available:

\begin{itemize}
\item using the current package instead,
\item retry finding the package (presumably after it has been created
  by the user),
\item specify a different package,
\item return an uninterned symbol, and
\item specify a symbol to return.
\end{itemize}

\noindent
For the \emph{triple colon} input, an error of type
\texttt{simple-reader-error} is signaled and no restart other than
\texttt{retry} and \texttt{abort} is available.

The \sbcl{} reader provides several extensions.  One such extension is
that a package prefix with two colon characters can precede any
expression, and not just a symbol.  The effect is that the reader then
uses the package named by the prefix as the default package to intern
symbols in the expression.  Another extension is that the exponent
marker \texttt{r} or \texttt{R} can be used to return a rational
number that is the exact value of the floating-point syntax used.  The
names of symbols created by the \sbcl{} reader can be normalized
according the ``Normalization Form KC'' which is specified for
Unicode.  Whether symbol names are normalized in this way is
controlled by the accessor \texttt{readtable-normalization} which
operates on a readtable.

The \sbcl{} reader can track source positions of each object returned.
Source tracking is handled through a special type of stream.  The
function \texttt{\%read-preserving-whitespace} checks for the stream
argument being of this type and, if so, registers the form and the
position of the form.  Two parallel adjustable vectors are used for
this purpose.

\subsection{Clozure Common Lisp (\ccl{})}

The \ccl{} reader is written in \cl{}, without the use of the specific
\clos{} features such as generic functions and standard classes.

For the \emph{incomplete} input, the \texttt{end-of-file} condition is
signaled and only restarts that return to previous break levels are
available.  For the \emph{missing package} input, several additional
restarts are available:

\begin{itemize}
\item supplying a different name for the package,
\item retry finding the package (presumably after it has been created
  by the user),
\item making the supplied package name a nickname for the package
  \texttt{common-lisp-user}.
\end{itemize}

\noindent
For the \emph{triple colon} input, no error is signaled at all, and
instead the third package marker is treated as escaped, so that it is
part of the symbol.

The \ccl{} reader can track source positions.  It registers the stream
position before and after an expression has been read, and creates an
instance of a \texttt{struct} named \texttt{source-note} and stores
this instance in a hash table.  Source positions are recorded only if
the stream is an element of the list
\texttt{*recording-source-streams*}, which is the case when the reader
is called for the purpose of evaluating the resulting expression.

\subsection{Clasp}

Clasp is a \cl{} implementation that was specifically designed to
interoperate with \cplusplus{} code, in order to take advantage of
existing \cplusplus{} libraries.  Clasp was originally based on \ecl{}
with significant changes for \cplusplus{} interoperability.  As such,
its reader was initially a \cplusplus{} conversion of the \ecl{}
reader.  However, in later bootstrapping stages, Clasp loads
\eclector{}, the subject of this paper, and makes it the default
reader.

There are plans to modify the bootstrapping procedure of Clasp so that
the initial \cplusplus{} reader is no longer needed.  For these
reasons, we do not review the \cplusplus{} reader of Clasp.
